# Cloud-Native Class Diagram Generator

## Project Path
`./CloudNativeClassDiagram`

## Dataset Used
FastAPI Official Full-Stack Template (Source Code)
Repository: `https://github.com/tiangolo/full-stack-fastapi-postgresql.git`
Path: `./sample_code/backend/app`

## Libraries Used
- PyTorch (`torch`)
- PyTorch Geometric (`torch-geometric`)
- NetworkX (`networkx`)
- Built-in Python Libraries: `ast`, `json`, `os`
- Mermaid.js (via VS Code Markdown Preview Extension)

## Brief Detail of Project
Replicated a research methodology to automatically generate modern class diagrams from Python source code using Graph Neural Networks (GNNs). 
We parsed the Abstract Syntax Tree (AST) of a real-world FastAPI backend to extract class relationships. We then implemented a custom "Cloud-Native" feature extractor to dynamically identify cloud and microservice components (like API routers and SQL models). A Graph Attention Network (GAT) processes the structure, and the final architecture is output as Diagrams-as-Code using Mermaid.js markdown.

## Observation
Relying purely on Python's ast module for graph extraction captures explicit structural relationships (like inheritance) but struggles with implicit, runtime dependencies such as Dependency Injection—which FastAPI uses heavily (e.g., Depends()). Furthermore, the current Graph Neural Network (GNN) relies on highly simplified node features (a scalar method count and a boolean is_cloud flag). A high-impact future enhancement would be to replace these basic features with dense vector embeddings generated by a Large Language Model processing the raw source code of each class. This would transition the GAT model from predicting relationships based purely on syntactic structure to predicting them based on semantic architectural intent.

### What to look for when you compare architecture.md and LLM-classdiagram.md:
When you put this side-by-side with your AST-generated `architecture.md`, look at the **arrows**. 
* Your AST diagram will likely have a massive web of arrows pointing to underlying framework classes (like `SQLModel` or `BaseModel`). 
* This LLM semantic diagram has arrows that describe **data flow**—showing exactly how an `ItemCreate` schema flows into the `ItemsRouter`, which is how software engineers actually think about system architecture.
